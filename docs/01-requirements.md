## 1. 概要

### 1.1 背景と目的

当社では SharePoint 上に経営資料・人事労務規程・営業関連文書を格納しているが、以下の課題がある。

- 社員が必要な情報にたどり着くまでに時間がかかる
- 経営層が意思決定時に関連文書を横断的に参照する手段がない
- 文書の閲覧権限が未設定で、機密文書が全社員に公開されている

これらを解決するため、SharePoint 文書を対象とした RAG チャットボットを PoC として構築し、実用性を検証する。

| 区分   | 内容                                                                      |
| ---- | ----------------------------------------------------------------------- |
| 主目的  | SharePoint 文書を対象に ACL 付き RAG チャットボットを PoC し、経営判断支援と社員問い合わせ削減の実現可能性を検証する |
| 技術習得 | チーム全員が RAG の設計・実装を本質から理解し、100-300名規模の案件で即戦力になる技術力を身につける                 |
| 副次効果 | SP × Azure RAG の構成が同規模の他社案件の提案テンプレートになる                                 |

### 1.2 スコープ

| 項目 | 値 |
|------|-----|
| 対象システム | SharePoint RAG チャットボット |
| 対象ユーザー | 10名（PoC）→ 100-300名（スケール時） |
| 対象データ | SharePoint 文書 100件以上（PDF, Office） |

### 1.3 ゴールとノンゴール

**ゴール（この PoC で達成すること）**
- GPT-RAG リファレンス準拠の RAG 基盤で、SP 文書の検索・回答精度を検証する
- 100-300名規模にスケール可能な設計パターンを確立する
- ACL（アクセス制御）を初期構築から組み込み、SP 権限連動の実現可能性を検証する

**ノンゴール（意図的にスコープ外とするもの）**
- 本番環境の構築（あくまで PoC であり、実運用は行わない）
- SharePoint 以外のデータソース対応（将来的には検討可能だが今回は対象外）
- モバイルアプリ / 社外向けサービス / マルチテナント
- UI/UX の最適化（検証用の最小チャット画面で十分）
- 高可用性構成（PoC では SLA 不要。スケール時の構成は設計書に記載）

## 2. ユースケース

| ID   | 名称         | アクター | 概要                                      |
| ---- | ---------- | ---- | --------------------------------------- |
| UC-1 | 経営判断支援     | 経営層  | 経営に関する質問に対し、経営層限定文書を含む全文書から根拠リンク付きで回答する |
| UC-2 | 社員問い合わせ削減  | 一般社員 | 社内制度・規程に関する質問に対し、閲覧権限のある文書のみから回答する      |
| UC-3 | 権限外アクセスの拒否 | 一般社員 | 権限外文書にのみ根拠がある質問に対し、情報を漏洩せず「該当なし」と応答する   |

### 2.1 回答の振る舞い

- 根拠がある場合: 自然言語で回答 + 根拠リンク1〜3件
- 根拠が不足する場合: 「該当する情報が見つかりませんでした」と応答（推測しない）
- 会話は継続可能（文脈を踏まえた連続質問に対応）

## 4. 機能要件

| ID   | 機能名     | 優先度 | 説明                       | 完了条件 |
| ---- | ------- | --- | ------------------------ | ---- |
| F-01 | 文書検索    | P0  | SharePoint 文書に対する自然言語検索  | 自然言語クエリに対し関連チャンクを返す |
| F-02 | 回答生成    | P0  | 検索結果を基にした根拠リンク付き回答の生成    | 根拠リンク付き回答を表示。根拠不足時は「該当なし」 |
| F-03 | 会話履歴    | P1  | セッション単位の会話履歴保存と文脈維持      | 直前の質問を踏まえた連続質問に文脈通りの回答 |
| F-04 | 文書取り込み  | P0  | SP 文書を検索可能な状態に取り込むパイプライン | 全フォルダの文書がインデックスに格納され検索可能 |
| F-05 | ユーザー認証  | P0  | Entra ID による SSO 認証      | Entra ID SSO でログイン可能 |
| F-06 | チャット UI | P0  | Web ベースのチャットインターフェース     | ブラウザでチャット入力・回答表示が可能 |
| F-07 | ACL 制御  | P0  | SP 権限に連動した文書レベルのアクセス制御   | 権限外文書が回答・チャンクに含まれない |

> **優先度定義**: P0 = PoC 成立に必須 / P1 = 重要だが PoC 初期は簡易版で可 / P2 = あれば望ましい

> 各機能の実現方式はアーキテクチャ設計書で定義する。

## 7. 成功基準と受入条件

### 7.1 成功基準（KPI）

| 軸          | PoC 目標                              |
| ---------- | ----------------------------------- |
| 検索精度       | 評価質問10問中 70% 以上が有効回答（ACL 考慮含む）    |
| アクセスコントロール | 権限漏洩 = 0件                           |
| コスト        | PoC 期間中の想定総額以内                      |

### 7.2 評価質問セット（暫定10問）

| #   | カテゴリ | 質問                  | 期待ソース       |
| --- | ---- | ------------------- | ----------- |
| 1   | 人事労務 | 情報セキュリティの基本方針は？     | 02_人事労務     |
| 2   | 人事労務 | プライバシーポリシーの内容は？     | 02_人事労務     |
| 3   | 人事労務 | 情報漏洩時の連絡フローは？       | 02_人事労務     |
| 4   | 経営   | 事業計画の概要は？           | 01_経営       |
| 5   | 経営   | 行動計画のサンプルを教えて       | 01_経営       |
| 6   | 営業   | GC株式会社との契約内容は？      | 03_営業       |
| 7   | 営業   | 機密保持契約の条件は？         | 03_営業       |
| 8   | Trap | （一般社員で）役員向け事業計画は？   | ACL で秘匿（01_経営） |

### 7.3 受入条件

- 評価質問10問中、7問以上で根拠リンク付きの有効回答
- 根拠不足時に推測せず「該当なし」と応答
- 応答時間 P95 で8秒以内
- 同一クエリで権限による結果分離。漏洩ゼロ
